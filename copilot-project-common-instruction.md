# Ver.1.0
# Copilot プロジェクト共通ルール・指示テンプレート

このファイルは、新規プロジェクトを立ち上げる際に Copilot に読ませることで、
**適切な README.md と .github/copilot-instructions.md を書くための共通ルール**を伝えるものです。

プロジェクト固有の技術スタックやドメインに関係なく、すべてのプロジェクトに適用されます。

---

## 1. プロジェクト開始時に必ず作るファイル

| ファイル | 目的 |
|---------|------|
| `README.md` | プロジェクトの全体像・方針・セットアップ手順・TODO。他のCopilotセッションがこれを読めばプロジェクトを把握できるように |
| `.github/copilot-instructions.md` | Copilotが守るべきルール。重要度が高い規則はここに集約 |
| `docs/system_config.md` | 全ファイルの一覧と役割、設定値・定数の管理台帳 |
| `docs/function_map.md` | 機能とファイルの対応表、依存関係、影響分析テンプレート |
| `.env.example` | 環境変数のテンプレート（.envはgit管理外） |
| `.gitignore` | 不要ファイルの除外設定 |

## 2. コーディング絶対ルール

### 2-1. パスの扱い
- ❌ **ファイル内で他ファイルのパスを直書き（ハードコード）禁止**
- ❌ **絶対パス禁止**（OS固有ツール（git bash等）からの呼び出しを除く）
- ✅ 設定モジュール経由、または `__file__` からの相対解決で参照する
- ✅ パスが必要な場合は設定ファイル（config）に定義し、そこから読む

### 2-2. マジックナンバー・定数
- ❌ **コード内にマジックナンバーを書かない**
- ✅ ユーザーが変更する値 → `.env` で管理
- ✅ アプリ内部の定数 → 設定モジュール or `docs/system_config.md` に記載
- ✅ 設定モジュールを通じてコードから参照する

### 2-3. ファイル管理
- 📋 ファイルを作成・削除したら **`docs/system_config.md`** を更新
- 📋 機能を追加・変更・削除したら **`docs/function_map.md`** を更新
- 🔍 変更時は **影響を受ける既存ファイル・機能がないかレビュー**

## 3. フォルダ構成ルール

### 3-1. クリーンアーキテクチャ思想
フォルダで責務を明確に分ける：

```
project/
├── api/        # ルーティング（エンドポイント定義のみ。薄く保つ）
├── services/   # ビジネスロジック
├── models/     # データ構造・型定義
├── core/       # 横断的関心事（設定、ログ、共通ユーティリティ）
├── db/         # データベース関連（リポジトリ層）
└── tests/      # テスト
```

### 3-2. rootに置けるファイル
rootには以下のみ：
- 実行ファイル（起動スクリプト）
- `.env` / `.env.example`
- `README.md`
- `setup.bat` / `setup.sh`（セットアップスクリプト）

その他のファイルは適切なフォルダに格納する。

### 3-3. クロスプラットフォーム対応
- コアロジックは **OS非依存** で書く
- OS固有の処理が必要な場合は **`windows/`** / **`mac/`** フォルダに分離
- 起動スクリプトは共通言語（Python等）で書くか、OS別に用意（`.bat` / `.sh`）

## 4. Git・コミットルール

- 📦 **機能単位で段階的にcommit**（一度に全部入れない）
- 📝 commitメッセージは内容を明確に（日本語OK）
- 🏷️ Copilotと作業した場合は `Co-authored-by` trailer を含める
- 🔀 ブランチ名の区切りは `_`（`/` ではなく）。例: `feature_add_chart`

---

## 5. ドキュメント更新タイミング

| トリガー | 更新対象 |
|---------|---------|
| ファイル追加・削除 | `docs/system_config.md` |
| 機能追加・変更・削除 | `docs/function_map.md` |
| 設計方針変更 | `README.md` + `.github/copilot-instructions.md` |
| 実装とドキュメントの乖離を発見 | 即時修正 |

**原則: 実装とドキュメントは常に同期させる。**

## 6. README.md に書くべき内容（テンプレート）

```markdown
# プロジェクト名

一行説明

## 概要
何をするアプリか、なぜ作るか

## 技術スタック
| レイヤー | 技術 |
|---------|------|
| ... | ... |

## アーキテクチャ
フォルダ構成図（tree形式）
各フォルダの責務説明

## プロジェクトルール
コーディングルール、命名規則、禁止事項

## セットアップ
OS別の手順（Windows / Mac / Linux）

## 起動方法
コマンド例

## TODO
未実装・予定機能のチェックリスト
```

## 7. copilot-instructions.md に書くべき内容（テンプレート）

```markdown
# Copilot Instructions for [プロジェクト名]

## プロジェクト概要
一行〜二行の説明

## 絶対守るルール（最重要）
1. パス直書き禁止（解決方法を具体的に記載）
2. マジックナンバー禁止（管理方法を記載）
3. ファイル作成・変更時のドキュメント更新義務
4. フォルダ構成ルール
5. rootに置けるファイルの制限

## 技術スタック
使用技術の列挙

## コミット規則
コミット粒度、メッセージ形式

## ドキュメント更新タイミング
何をしたら何を更新するか
```

## 10. Copilotへの追加指示

- 🔍 作業前に `README.md` と `docs/` を読んでプロジェクトを把握する
- 🧪 変更後は既存機能が壊れていないか確認する
- 📖 不明点があればユーザーに質問する（勝手に判断しない）
- 🔄 rubber-duck agent や code-review agent を活用してレビューする
- ⚠️ セキュリティ（APIキー、トークン）はログや公開コードに含めない
